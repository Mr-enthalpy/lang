# v0.8 Meta Construction Value / Pattern Clarifications

**Status: Draft semantic reset / addendum for v0.8-adjacent meta-construction work.
Not current public language behavior. This document refines the construction
constraints in `v0.8-meta-construction-agent-constraints.md`; it does not replace
that document.**

For ordinary generic type-style meta construction, this document overrides any
older phrasing that describes meta invocation result as a bare `TypeValueId`
reduction.

## 1. Purpose

This document fixes several semantic boundaries that must be preserved before
implementing ordinary type-to-type meta-functions such as:

```text
(T)Vec::std
(T)Option::std
(A, B)Pair::std
```

The central point is that a type-to-type meta-function invocation does not merely
reduce to a `TypeValueId`. It returns a value. That value may be a type-rank
value, a construction value, an extraction-facing value, or a forwarded existing
value, depending on how the meta body binds and returns its result.

Therefore the implementation must distinguish:

```text
computed type value
constructed external value
forwarded existing value
declaration binding target
extraction-facing pattern interface
```

Collapsing these into a single `TypeValueId` is only acceptable for a narrow
placeholder proof path. It is not the semantics of ordinary v0.8 type-to-type
meta construction.

## 2. Type constructors are values

An expression such as:

```text
(uint8)Vec::std
```

is a value. Its value is the construction expression itself:

```text
value((uint8)Vec::std) = (uint8)Vec::std
```

This is consistent with ordinary type values:

```text
let t = uint8
value(t) = uint8

let t = (uint8)Vec::std
value(t) = (uint8)Vec::std
```

`(uint8)Vec::std` is simultaneously:

```text
a value
a type-rank value
a construction result
material that may expose an extraction interface
```

The implementation must not treat this as a mere erased alias for its argument.
A generated type constructor value is not automatically equal to the argument
from which it was computed.

## 3. Call, construction, and extraction

Construction and extraction may be isomorphic. A value produced by a constructor
may expose an extraction interface that recognizes the corresponding constructed
shape.

Call and extraction are not isomorphic. A call returns a value. Extraction works
on the returned value's static pattern space, not on the call target, not on the
call history, and not on the internal computation path of the callable.

The following distinction must be preserved:

```text
call          -> returns a value
construction  -> gives the returned value a construction shape
extraction    -> operates on the returned value's extraction-facing pattern space
```

A call may return a construction. But not every call result is extractable
through the call's surface name, and not every call history is an extraction
shape.

## 4. Meta bodies use ordinary block return semantics

A meta-function body is an ordinary block for result-consumption purposes.

The body returns by the same rule as every other block: an otherwise unconsumed
final expression is the current block result. Therefore:

```lang
meta+ runtime let id = (self, t: type): meta -> r: type => {
    r = t;
    r;
};
```

does not use a special meta-return form. The final expression `r` is simply the
result of the meta-function body block.

The same applies to:

```lang
meta+ runtime let id = (self, t: type): meta -> r: type => {
    r === t;
    r;
};
```

The difference between the two functions is not return control flow. The
difference is the value semantics of the return slot:

```text
r = t    => generated construction value
r === t  => forwarded existing value
```

Thus the meta-function case is only an instance of the general invocation and
block-result rules.

## 5. Symbol shielding

Ordinary generative meta-functions have symbol shielding.

For a generative return:

```text
r = t
```

The externally visible value-construction-extraction expression is determined by:

```text
callee symbol + canonical argument product + build/config identity
```

not by:

```text
internal temporary name
internal computation path
source spelling of intermediate expressions
```

For a forwarding return:

```text
r === t
```

The returned value is the forwarded target. The call produces a value by
forwarding an existing construction / value identity rather than by exposing a
new generative construction identity.

A minimal implementation model is:

```text
r = t
  => GeneratedConstructionValue {
       callee,
       canonical_args,
       build_identity,
       computed_material = t,
       external_shape = (args)callee,
     }

r === t
  => ForwardedValue {
       target = t,
     }
```

A `MetaReductionResult::TypeValue(TypeValueId)` can model only a narrow
forwarding-like placeholder. It cannot represent the full distinction required
for v0.8 ordinary generative type constructors.

## 6. Return slot names are internal

This is invalid:

```lang
meta+ runtime let id = (self, t: type): meta -> t => {
    t;
};
```

The parameter name `t` and the return slot name `t` collide. The return slot must
have its own binding name. The external identity is determined by generative
shielding or forwarding semantics.

## 7. Rank annotation after symbol binding

After symbol binding, rank annotations split into two semantic classes.

### 7.1 Hole-bearing rank annotations are extraction forms

Example:

```lang
let <T: type> val: T Vec::std = ...
```

Here the annotation contains a hole name. The annotation is therefore an
extraction expression over the rank annotation. It belongs to the extraction side
of the call / construction / extraction correspondence.

The annotation attempts to extract `T` from the rank/type structure of `val` by
using the extraction interface exposed by the constructed value.

This is not merely a Boolean assertion. It is a binding extraction over a static
rank pattern.

### 7.2 Hole-free rank annotations are assertions

Example:

```lang
let val: uint8 Vec::std = ...
```

Here the annotation has no hole name. The annotation is therefore a rank
assertion of the form:

```text
target : annotation
```

The judgment evaluates whether the target satisfies the annotation:

```text
target : annotation == true   if the annotation holds
target : annotation == false  if the annotation does not hold
```

In this form, `annotation` belongs to the call side of the call / construction /
extraction correspondence, not to the extraction side.

Thus the following two surface forms must not be normalized into the same
semantic object:

```lang
let <T: type> val: T Vec::std = ...
let val: uint8 Vec::std = ...
```

The first is extraction-bearing. The second is assertion-bearing.

## 8. Explicit value-to-pattern bridge

A value is not a pattern. A pattern is not a value.

A value enters pattern space only through an explicit bridge, such as postfix
`?` or a later explicit value-to-pattern operation.

This matters for generated type constructor values:

```text
(uint8)id
```

may be a value, but extraction does not automatically inspect the call history.
Extraction sees only the value's exposed static pattern space.

If a forwarding meta-function returns `uint8`, then attempting to extract it as
an `id` construction may fail for the simplest reason:

```text
(uint8)id == uint8
```

The returned construction is `uint8`, not an `id`-shaped construction.

## 9. Single and multiple returns

Multi-return is not a separate return mechanism. A function returns one value.
That value may be an expression value or a product value.

The structural distinction is:

```text
e      -> e
(e, e) -> P
```

Returning one parameter returns an expression value:

```text
return e => e
```

Returning multiple parameters returns a product:

```text
return (e1, e2) => P(e1, e2)
```

Any later decomposition is extraction over the returned value.

## 10. Product extraction after return

For a single-return function:

```lang
let t = () |> single_return;
```

The result is:

```text
t == ()single_return
```

The following is an error if the value does not expose a product extraction
interface:

```lang
let a, b = t?;
```

The trivial value-to-pattern bridge preserves the value:

```text
t? == t
```

For a two-return function:

```lang
let t = () |> two_return;
```

The result is still the call construction value:

```text
t == ()two_return
```

But if the returned value exposes a product extraction interface, then:

```lang
let a, b = t?;
```

is valid, and:

```text
t? != t
(a, b) == t?
(a, b) != t
```

This follows from:

```text
e? -> e
P? -> e, e
```

The implementation must not treat multi-return as a hidden tuple unpacking rule
at the call boundary. The call returns one value; extraction may later decompose
that value.

## 11. Implementation requirements

Before ordinary v0.8 type-to-type meta construction can be called implemented,
the meta invocation result layer must be able to distinguish at least:

```text
GeneratedConstructionValue
ForwardedValue
Residual / not-yet-reduced value
Diagnostic
```

A future `MetaExpansionResult` or equivalent binding layer must also keep these
roles distinct:

```text
replacement / returned value
namespace delta to install
declaration binding target
extraction-facing pattern interface
return-object policy
```

The following collapse is forbidden outside narrow placeholder tests:

```text
GeneratedConstructionValue(args, callee) -> TypeValueId(argument)
```

The following shortcut is also forbidden as the semantics of ordinary generated
constructors:

```text
r = t behaves like r === t
```

A primitive identity proof path may temporarily demonstrate graph-resolved
meta-call plumbing, candidate preparation, or cache-key formation. It must not be
used as evidence that ordinary generative type-to-type meta construction has
been implemented.

## 12. Minimal test obligations

Before `(T)Vec::std` or `(A, B)Pair::std` is implemented, add a smaller pair of
semantic tests.

Generative identity-like constructor:

```lang
meta+ runtime let IdBox = (self, t: type): meta -> r: type => {
    r = t;
    r;
};

let T: type = (uint8) |> IdBox;
```

Expected:

```text
T == (uint8)IdBox
T != uint8
```

Forwarding identity:

```lang
meta+ runtime let IdAlias = (self, t: type): meta -> r: type => {
    r === t;
    r;
};

let T: type = (uint8) |> IdAlias;
```

Expected:

```text
T == uint8
(uint8)IdAlias == uint8
```

Product-return distinction:

```lang
let t = () |> single_return;
let a, b = t?;        // ERROR unless t exposes product extraction

let p = () |> two_return;
let a, b = p?;        // valid if p exposes product extraction
```

These tests directly protect the intended semantics without requiring the full
`Vec`, `Option`, or `Pair` constructor family to exist first.
