# v0.8 Symbolic Construction Values and Extraction Interfaces

**Status: Draft semantic specification for v0.8-adjacent meta-construction work.
Not current public language behavior.**

This document depends on the extraction-view model defined in
`spec/design/patterns-overload/return-value-extraction-and-implicit-decomposition.md`.

The v0.8 semantic spine is defined in
`spec/design/v0.8-semantic-spine.md`. This document is the detailed
construction-value chapter of that spine.

Three axioms:

1. An evaluation returns one result object whose normal form is `e` or `P`.
2. A type-constructor call returns a symbolic construction value.
3. Extraction is a view transition, not call-history inversion.

## 1. Purpose

A type-to-type meta-function invocation returns an invocation value. That value
may be a type-rank value, a construction value, an extraction-facing value, or a
forwarded existing value, depending on how the meta body binds and returns its
result.

Therefore the implementation must distinguish:

```text
computed type value
constructed external value
forwarded existing value
declaration binding target
extraction-facing pattern interface
```

`TypeValueId` is only a derived first-order type-value projection material. It
is not an invocation result, canonical identity material, binding source,
construction identity, or type-definition identity.

## 2. Type Constructors Are Values

An expression such as:

```text
(uint8)Vec::std
```

is a value. Its value is the construction expression itself:

```text
value((uint8)Vec::std) = (uint8)Vec::std
```

This is consistent with ordinary type values:

```text
let t = uint8
value(t) = uint8

let t = (uint8)Vec::std
value(t) = (uint8)Vec::std
```

`(uint8)Vec::std` is simultaneously:

```text
a value
a type-rank value
a construction result
material that may expose an extraction interface
```

The implementation must not treat this as a mere erased alias for its argument.
A generated type constructor value is not automatically equal to the argument
from which it was computed.

## 3. Call, Construction, and Extraction

Construction and extraction may be isomorphic. A value produced by a constructor
may expose an extraction interface that recognizes the corresponding constructed
shape.

Call and extraction are not isomorphic. A call returns a result object whose
normal form is `e` or `P`. Extraction works on the returned value's static
pattern space, not on the call target, not on the call history, and not on the
internal computation path of the callable.

The following distinction must be preserved:

```text
call          -> returns a result object with normal form e or P
construction  -> gives a non-product returned value a construction shape
extraction    -> operates on the returned value's extraction-facing pattern space
```

A call may return a construction. A call may also return product normal form.
Neither case licenses extraction through call history.

## 4. Meta Bodies Use Ordinary Block Return Semantics

A meta-function body is an ordinary block for result-consumption purposes.

The body returns by the same rule as every other block: an otherwise unconsumed
final expression is the current block result. Therefore:

```lang
meta+ runtime let id = (self, t: type): meta -> r: type => {
    r = t;
    r;
};
```

does not use a special meta-return form. The final expression `r` is simply the
result of the meta-function body block.

The same applies to:

```lang
meta+ runtime let id = (self, t: type): meta -> r: type => {
    r === t;
    r;
};
```

The difference between the two functions is not return control flow. The
difference is the value semantics of the return slot:

```text
r = t
  -> GeneratedConstructionValue or GeneratedTypeDefinitionValue, depending on returned material

r === t
  -> ForwardedValue(TypeSymbol)
```

Thus the meta-function case is only an instance of the general invocation and
block-result rules.

## 5. Symbol Shielding

Ordinary generative meta-functions have symbol shielding.

For a generative return:

```text
r = t
```

The externally visible value-construction-extraction expression is determined by:

```text
callee symbol + canonical argument product + build/config identity
```

not by:

```text
internal temporary name
internal computation path
source spelling of intermediate expressions
```

For a forwarding return:

```text
r === t
```

The returned value is the forwarded target. The call produces a value by
forwarding an existing construction / value identity rather than by exposing a
new generative construction identity.

A minimal implementation model is:

```text
r = t
  -> GeneratedConstructionValue {
       callee,
       canonical_args,
       build_identity,
       computed_material = t,
       external_shape = (args)callee,
     }

r = type definition material
  -> GeneratedTypeDefinitionValue {
       callee,
       canonical_args,
       field_signature_material,
       type_definition_id,
     }

r === t
  -> ForwardedValue {
       target = TypeSymbol(t),
     }
```

## 6. Return Slot Names Are Internal

This is invalid:

```lang
meta+ runtime let id = (self, t: type): meta -> t => {
    t;
};
```

The parameter name `t` and the return slot name `t` collide. The return slot must
have its own binding name. The external identity is determined by generative
shielding or forwarding semantics.

## 7. Rank Annotation After Symbol Binding

After symbol binding, rank annotations split into two semantic classes.

### 7.1 Hole-Bearing Rank Annotations Are Extraction Forms

Example:

```lang
let <T: type> val: T Vec::std = ...
```

Here the annotation contains a hole name. The annotation is therefore an
extraction expression over the rank annotation. It belongs to the extraction side
of the call / construction / extraction correspondence.

The annotation attempts to extract `T` from the rank/type structure of `val` by
using the extraction interface exposed by the constructed value.

This is not merely a Boolean assertion. It is a binding extraction over a static
rank pattern.

### 7.2 Hole-Free Rank Annotations Are Assertions

Example:

```lang
let val: uint8 Vec::std = ...
```

Here the annotation has no hole name. The annotation is therefore a rank
assertion of the form:

```text
target : annotation
```

The judgment evaluates whether the target satisfies the annotation:

```text
target : annotation == true   if the annotation holds
target : annotation == false  if the annotation does not hold
```

In this form, `annotation` belongs to the call side of the call / construction /
extraction correspondence, not to the extraction side.

Thus the following two surface forms must not be normalized into the same
semantic object:

```lang
let <T: type> val: T Vec::std = ...
let val: uint8 Vec::std = ...
```

The first is extraction-bearing. The second is assertion-bearing.

## 8. Explicit Value-To-Pattern Bridge

A value is not a pattern. A pattern is not a value.

A value enters pattern space only through an explicit bridge, such as postfix
`?` or a later explicit value-to-pattern operation.

This matters for generated type constructor values:

```text
(uint8)id
```

may be a value, but extraction does not automatically inspect the call history.
Extraction sees only the value's exposed static pattern space.

If a forwarding meta-function returns `uint8`, then attempting to extract it as
an `id` construction may fail for the simplest reason:

```text
(uint8)id == uint8
```

The returned construction is `uint8`, not an `id`-shaped construction.

## 9. Product Return Normal Form

Product return is not a separate return mechanism and it is not an outer wrapper
around a product. An evaluation returns one result object whose normal form may
be `e` or `P`.

The structural distinction is:

```text
e      -> ordinary non-product value point
P      -> product value / product normal form
```

Returning one expression value returns `e`:

```text
return e => e
```

Returning a product returns `P` directly:

```text
return (e1, e2) => P(e1, e2)
```

For:

```lang
let P = () |> two_return;
```

if `two_return` returns product normal form `(a, b)`, then:

```text
P == (a, b)
P? == P
let a, b = P
(a, b) == P
```

For a non-product value `e`:

```text
e? == e             // if e is leaf
let a, b = e?       // error if e? is still leaf
(a, b) == e         // false if e is not product normal form
```

For a non-leaf construction value `e` with an exposed product view:

```text
e? == P
P == e?             // true
P == e              // false
P A== e             // true under named constructor / pattern A
```

`A==` is provisional notation for constructor/pattern mediated equality. It is
not ordinary value equality.

The following invariants are contract-level:

```text
1. `?` does not inspect call history;
2. `?` is not recursive — it peels one declared top extraction pattern layer;
3. product normal form satisfies P? = P;
4. binding context may request one implicit `?` when the left pattern demands extraction;
5. value equality never inserts `?`;
6. named extraction is not bare product extraction;
7. product matching consumes P in the pattern matcher;
8. only the selected branch of a guarded sum-pattern expression may perform
   lookup, policy check, meta invocation, or local symbol construction;
   unselected branches have no obligation of any kind.
```

## 11. Implementation Requirements

Before ordinary v0.8 type-to-type meta construction can be called implemented,
the meta invocation result layer must be able to distinguish at least:

```text
ForwardedValue(TypeSymbol)
GeneratedConstructionValue
GeneratedTypeDefinitionValue
Diagnostic
```

A binding layer must also keep these roles distinct:

```text
replacement / returned value
namespace delta to install
declaration binding target
extraction-facing pattern interface
return-object policy
```

The following collapse is forbidden:

```text
GeneratedConstructionValue(args, callee) -> TypeValueId(argument)
```

The following shortcut is also forbidden as the semantics of ordinary generated
constructors:

```text
r = t behaves like r === t
```

A primitive identity proof path may demonstrate graph-resolved meta-call
plumbing, candidate preparation, or cache-key formation. It must not be used as
evidence that ordinary generative type-to-type meta construction has been
implemented.

## 12. Minimal Test Obligations

Before `(T)Vec::std` or `(A, B)Pair::std` is implemented, add a smaller pair of
semantic tests.

Generative identity-like constructor:

```lang
meta+ runtime let IdBox = (self, t: type): meta -> r: type => {
    r = t;
    r;
};

let T: type = (uint8) |> IdBox;
```

Expected:

```text
T == (uint8)IdBox
T != uint8
```

Forwarding identity:

```lang
meta+ runtime let IdAlias = (self, t: type): meta -> r: type => {
    r === t;
    r;
};

let T: type = (uint8) |> IdAlias;
```

Expected:

```text
T == uint8
(uint8)IdAlias == uint8
```

Product-return distinction (see also
`spec/design/patterns-overload/return-value-extraction-and-implicit-decomposition.md`):

```lang
let e = () |> single_return;
let a, b = e?;        // ERROR unless e? exposes product normal form
let a, b = e;         // ERROR if pattern-directed repair cannot expose product normal form

let P = () |> two_return;
let a, b = P;         // valid: P is product normal form
P? == P;              // true
(a, b) == P;          // true when P is exactly (a, b)
```

These tests directly protect the intended semantics without requiring the full
`Vec`, `Option`, or `Pair` constructor family to exist first.

## 13. Return-Normal-Form Substrate

The implementation substrate for this contract is shape-level:

```text
EvalResultNormalForm = ValuePoint(e) | Product(P)
```

`question_view` follows the documented view-transition rules:

```text
Product(P) -> Product(P)
leaf e     -> e
non-leaf e -> exposed view
```

Binding shape matching is the only phase in this substrate that may report a
one-step extraction repair:

```text
ProductPattern + P -> Direct
ProductPattern + non-leaf e exposing P -> AfterExtraction
ProductPattern + leaf e -> Mismatch
```

A generated struct type records instance extraction metadata as a named field
product with value projections. The type-definition invocation value remains a
pure invocation value; declared symbols, associated namespaces, field
projections, and instance extraction metadata are binding materialization
artifacts.

## 14. core::struct Normalized-AST Decoding Contract

`core::struct` reads generic normalized AST and performs struct-specific
interpretation locally. The parser and normalizer are not modified for struct
syntax.

Contract-level invariants:

```text
1. The struct decoder (`struct_decoder.rs`) owns struct-specific interpretation
   of generic NormExpr nodes.
2. No parser or normalizer node is introduced for struct type-pattern
   expressions. The normalized AST remains generic.
3. `|` is the canonical sum-pattern result form accepted by the decoder.
   `+` is a pattern-combination reduction action and is not directly accepted
   as a canonical sum by the decoder.
4. The decoder produces TypePatternExprShape, which is attached to
   GeneratedTypeDefinitionValue.
5. The decoder does not install NamespaceDelta. Only binding/materialization
   installs graph objects.
6. Leaf type expressions are not restricted to simple paths. The final bare
   Name in an application chain is the local field/payload name; the prefix is
   the type expression. This rule is struct-decoding-local only.
```
