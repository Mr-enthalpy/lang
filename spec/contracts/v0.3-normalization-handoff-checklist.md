# v0.3 Normalization Handoff Checklist

## 1. Purpose

This document was the handoff checklist used to enter v0.3. It records the
checklist for entering `v0.3 — Normalized AST Specification`.

It does not define:

- Normalized AST node set
- Raw AST → Normalized AST lowering algorithm
- name resolution
- type checking
- kind checking
- operator lookup
- alias target resolution
- canonical matching
- closure materialization
- ownership / NLL / drop
- HIR / MIR
- interpreter behavior
- code generation

## 2. Stage boundary

```text
v0.1   — Raw AST Frontend — completed
```

## 4. v0.3 may assume

v0.3 Normalized AST Specification work may assume:

- The v0.2 public frontend specification set is the source-level syntax authority.
- The frozen Raw AST surface is complete for v0.2.
- Source text normalization is LF-based.
- Spans refer to normalized source text.
- Lexer token categories are frozen.
- The lexer is weak: names are not keyword-classified.
- Program forms use hard-only boundaries: semicolon, right brace, EOF.
- Raw AST is source-preserving and non-desugared.
- Operator sugar is preserved as Raw AST.
- Product and product-extract architecture is frozen.
- Pipe / segment / operator-expression architecture is frozen.
- `BindingSlotAst` is the shared raw binding-slot shape.
- Alias-let is preserved as `let binder === EntityRef`.
- Alias RHS `EntityRef` is parser-preserved but unresolved.
- Closure AST is preserved but not materialized.
- Canonical skeletons and deduce lists are parser-preserved but not matched.
- Diagnostics and `ErrorAst` nodes are available for recovery-aware normalization.
- Normalized AST is expected to be desugared but still non-semantic.

## 5. v0.3 must not assume

v0.3 Normalized AST Specification work must not assume:

- Names are resolved.
- Operators are resolved.
- Operator declarations exist.
- Alias targets are resolved.
- Types are checked or inferred.
- Kinds are checked or inferred.
- Canonical skeletons are admitted.
- Deduce-list names are validated.
- Binding patterns are semantically admissible.
- Closure captures are analyzed.
- Closures are materialized as callable objects.
- `with { ... }` carries dependency, lifetime, or ownership semantics.
- `return`, `else`, `match`, `effect`, `sync`, `drop`, `move`, `guard`, or
  similar names have semantic force merely because they appear as names.
- Parser acceptance implies semantic validity.
- Diagnostics are semantic errors.

## 6. Required Raw AST input families

Every Raw AST construct family the normalization pass must consume is listed
below. Obligations are abstract — this document does not define the final
Normalized AST node types.

| Input family | v0.3 obligation | Must preserve? | Must desugar? | Must not do |
|---|---|---|---|---|
| Source normalization and spans | Normalized AST spans must be traceable to normalized source text. | Yes. | No. | Do not lose source origin. |
| Lexer token categories | Accept the frozen token stream. | Preserve token text. | No. | Do not re-lex or re-classify tokens. |
| Weak lexer rule | Accept `Name` tokens without keyword classification. | Yes. | No. | Do not introduce keyword tokens. |
| Operator spellings | Accept operator sugar as Raw AST. | Preserve operator spelling. | Yes, in normalization. | Do not perform operator lookup or overload resolution. |
| Program / form boundaries | Accept hard-only form boundaries. | Preserve form order. | No. | Do not introduce line-based form separation. |
| FormAst family | Accept `Let`, `AliasLet`, `Expr`, `Error`. | Preserve form structure. | Yes, into normalized declaration/expression. | Do not reorder forms. |
| BindingSlotAst | Accept the shared binding-slot shape. | Preserve slot fields. | Yes, into normalized declaration/pattern. | Do not validate policy or annotation semantics. |
| BindingPatternAst | Accept `Binder`, `Product`, `Skeleton`, `Error`. | Yes. | Yes, into normalized pattern form. | Do not judge pattern admissibility. |
| WithClauseAst | Accept `with { }` and `with { name, ... }`. | Yes. | Preserve the payload; desugar wrapping only if needed. | Do not resolve with-clause names. |
| ProductExprAst / ProductExtractAst | Accept product expressions and product extraction. | Preserve unit elements. | Yes, into normalized product/call structure. | Do not assign source/insert/right-target roles. |
| Pipe expressions and pipe segments | Accept pipe architecture as outer skeleton. | Preserve segment order. | Yes, into explicit call structures. | Do not interpret pipe as monadic bind or control flow. |
| Pipe branch-name shorthand | `\|> name { ... }` is mechanically equivalent to `\|> (_ name) { ... }`. | Already desugared in Raw AST. | No further desugar needed. | Do not generalize into branch-arm sugar family. |
| OperatorExpr / OperatorSugar | Accept prefix-negative, postfix, and binary sugar. | Preserve operator spelling and fixity. | Yes, to named operator calls. | Do not perform operator lookup or validation. |
| Prefix negative | `-x` preserved as `OperatorSugar { fixity: Prefix, operator: "-" }`. | Preserve source span and diagnostic origin. | Yes, to typed-zero binary subtraction. Exact normalized form specified in v0.3. | Do not resolve it as an overloadable prefix operator. |
| Atom suffixes | Accept `::`, `.`, `..`, `[...]` sugar. | Preserve source order. | Yes, to normalized suffix-call forms. | Do not perform field lookup or method dispatch. |
| Navigation | Accept inner-to-outer navigation. | Preserve component order. | Yes, into normalized navigation form. | Do not resolve navigation components. |
| Member sugar | Accept `object.field`. | Yes. | Yes, to normalized member-access form. | Do not resolve field identity. |
| Double-dot sugar | Accept `object..method(args)`. | Yes. | Yes, to normalized method-call form. | Do not resolve method identity. |
| Bracket-call sugar | Accept `obj[args...]`. | Preserve operator identity. | Yes, to operator-call form with `[]` spelling. | Do not interpret as indexing, slicing, or container access. |
| Closure AST | Accept `InPlace` and `Explicit` closures. | Preserve closure structure. | Preserve head/body structure; no materialization. | Do not materialize into callable objects. |
| Closure head clauses | Accept deduce, capture, params, return, and head clause tail. | Preserve clause order and shape. | Yes, into normalized head structure. | Do not validate contract/lifetime/predicate clauses. |
| Canonical skeletons | Accept parser-preserved skeletons. | Preserve skeleton shape. | Yes, into normalized pattern forms. | Do not execute matching or evaluate canonical forms. |
| Deduce lists | Accept deduce lists in strong binding contexts. | Preserve binder declarations. | Yes, into normalized hole/pattern structure. | Do not validate or resolve deduce-list names. |
| Alias-let | Accept `let binder === EntityRef`. | Preserve binder and target. | Preserve as unresolved normalized declaration. | Do not resolve alias target. |
| Alias RHS EntityRef | Accept EntityRef in alias RHS. | Preserve navigation components. | Preserve as unresolved entity reference. | Do not resolve EntityRef to a semantic entity. |
| ErrorAst | Accept ErrorAst recovery markers. | Preserve span and message. | Preserve for diagnostic rewiring. | Do not use ErrorAst as a typed hole or proof obligation. |
| Diagnostics | Accept the 29-code diagnostic surface. | Preserve code, message, span. | Rewire spans through normalization where needed. | Do not add or remove Raw AST frontend DiagnosticCodes. |
| Dumps and golden snapshots | Accept stable dump format as frozen contract. | Preserve golden expectations. | Add normalized dumps only under explicit v0.4 spec. | Do not change existing golden snapshots. |

## 7. Required diagnostic and recovery inputs

v0.3 must be designed with diagnostics and recovery in mind. Normalization
must preserve enough origin information to support later diagnostic
attribution.

Required input surface:

- `DiagnosticCode` — 29 variants across lexer, parser, operator, and alias categories
- `Diagnostic` — carries `code`, `message`, and `span`
- `ErrorAst` — recovery markers in 11 AST positions
- Lexer diagnostics (4): `InvalidToken`, `UnclosedString`, `UnclosedComment`, `InvalidNumericLiteral`
- Parser diagnostics (17): 16 guaranteed + 1 optional (`UnusedClosureAst`)
- Operator diagnostics (3): guaranteed
- Alias diagnostics (5): 4 guaranteed + 1 reserved (`InvalidAliasBinder`)

Diagnostic and recovery boundary:

- Do not add or remove `DiagnosticCode` variants.
- Do not change diagnostic trigger conditions or recovery behavior.
- v0.3 must not mutate the frozen Raw AST frontend diagnostic code set.
- Any normalization-stage diagnostics require an explicit v0.3 diagnostic specification.

## 8. Required normalization obligations

The following obligations must be decided or resolved during v0.3
specification. This checklist does not prescribe the answers — it enumerates
what must be on the v0.3 agenda.

- Decide how Raw AST forms map into normalized declaration/expression structures.
- Decide how ordinary let and alias-let are represented before name resolution.
- Decide how `BindingSlotAst` becomes normalized declaration/pattern structure.
- Decide how product expressions and product extraction become normalized structures.
- Decide how pipe expressions become explicit call/composition structures.
- Decide how right-target subsegments are represented.
- Decide how operator sugar becomes named operator-call structure.
- Decide how bracket-call sugar becomes an operator-call-like structure.
- Decide how member sugar, double-dot sugar, and navigation are represented.
- Decide how closure literals and closure heads are represented without closure materialization.
- Decide how canonical skeletons and deduce lists are preserved before matching.
- Decide how generated/desugared nodes carry source origin.
- Decide how diagnostics are rewired through desugaring.

Each item remains a checklist obligation. This document does not resolve them.

## 9. Open v0.3 design questions

The following questions from `spec/planning/open-questions.md` become active
during v0.3–v0.5. They are not resolved by this document.

- **N-AST-1.** Exact Normalized AST node set. What are the exact node types?
  Candidates: normalized call, normalized pattern, normalized declaration.
  Should there be a single unified expression node or distinct per-form nodes?

- **N-AST-2.** Whether Normalized AST lives in `lang_syntax` or a new crate
  (e.g., `lang_norm`).

- **N-AST-3.** Whether raw-to-normalized dumps should be golden-tested.

- **N-AST-4.** How to represent symbolic builtins introduced by desugaring
  (e.g., `operator::call`, `member::lookup`, `pattern::bind`).

- **N-AST-5.** How to preserve source origins through desugaring. Desugaring
  creates new AST nodes that did not appear in source text. How should source
  spans and diagnostic attribution be preserved?

- **N-AST-6.** Whether right-target subsegments become nested call nodes.
  Right-target subsegments (`f (a) g`) are currently flat in Raw AST.

- **N-AST-7.** How to represent pattern normalization for let, params, returns,
  and canonical skeletons.

- **N-AST-8.** How to represent alias declarations before name resolution.
  Alias bindings reference compile-time entities that are not yet resolved.
