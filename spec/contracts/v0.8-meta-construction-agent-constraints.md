# Agent Construction Constraints: Build / Symbol / Pattern / Product / Policy / Meta Invocation Closed-Loop Progress

**Status: Draft construction contract for v0.8-adjacent implementation work.
Not current public language behavior. Not a replacement for existing public,
design, planning, or implementation specs.**

This document is not a new language design overview and does not replace any
existing specification text. It is a construction-constraint document for future
engineering agents working around v0.8, especially on ordinary generic
type-style meta-function capability such as `Vec(T)`, `Option(T)`, and
`Pair(A, B)` type-to-type meta-functions.

Its core purpose is to prevent agents from implementing a local feature through
an ad hoc bypass that happens to work in isolation, while damaging the shared
semantic foundation across build/package, symbol-world, patterns-overload,
product normalization, policy-capability, meta-invocation, and mechanical
lowering.

The project takes a long-term implementation approach. Implementation may be
staged, use placeholders, or use stubs, but object boundaries and semantic
direction must not be rewritten for short-term convenience.

## 1. General Principle: Advance the Closed Loop, Not Isolated Points

Any v0.8-related implementation task must not make progress from a single module
alone. For example:

- Do not create a special execution path only for `struct`.
- Do not recognize `Vec(T)` through a name check or AST-shape check.
- Do not temporarily parse arguments only for one generic type constructor.
- Do not let the type checker, meta interpreter, policy checker, and resolver
  maintain parallel symbol tables.
- Do not let product argument reading bypass pattern / argument-shape objects.
- Do not make policy apply only at lookup time or only at execution time.
- Do not let build/package identity disappear from candidate identity, meta
  instance keys, provenance, or cache keys.

Every implementation point must be checked against this loop:

```text
build/package identity
  -> namespace graph / SymbolObject
  -> SymbolId / PlaceId / TypeValueId / AliasChain
  -> ProductObject / ArgProductShape / RawArgShape / ParameterShape
  -> policy-aware candidate lookup
  -> body-entry / return-object policy checking
  -> formal meta invocation
  -> MetaReductionResult / MetaExpansionResult
  -> NamespaceDelta atomic install
  -> generated instance identity / provenance / diagnostics
```

If one stage is not fully implemented yet, use a placeholder. The placeholder
must live in the correct object slot; it must not create a new bypass semantic.

## 2. No-Bypass Rule

Every component that needs symbols must go through the shared namespace graph
world model.

This includes, but is not limited to:

```text
resolver
early meta-function lookup
struct
assert
verify
generic type meta-functions
type checker
policy checker
seal stage
IDE index
cache layer
diagnostics
later HIR lowering
```

Forbidden:

```text
parser special-cases struct / Vec / Pair
normalizer special-cases struct / Vec / Pair
meta executor searches raw string names outside resolver
type checker maintains private global symbol map
policy checker re-resolves by string path
generic instantiation key uses source text only
core symbols are globally visible through ambient state rather than explicit graph mount
```

The only permitted bootstrap magic is a compiler-seeded core package artifact,
a default core mount, and built-in payload placeholders. Even then, those symbols
must be installed into the namespace graph and returned by the resolver as
ordinary `SymbolObject`s.

## 3. Target Stage: Ordinary Generic Type-Style Meta-Functions

The current target is not full metaprogramming, full runtime overload,
value-to-value compile-time execution, a full type checker, or a full policy
lattice.

The target is the earliest restricted type-to-type meta construction:

```text
type -> type
type x type -> type
single entry / single exit
no intermediate control flow
meta body is Normalized AST
construction is graph / delta / policy governed
```

Typical targets:

```text
Vec(T)
Option(T)
Pair(A, B)
```

These targets must be implemented as ordinary graph-resolved meta callables, not
compiler special cases.

A valid generic type-style meta-function invocation should follow this flow:

```text
resolve callee symbol through namespace graph
collect callable candidates under lookup policy
form ProductObject / ArgProductShape
form per-element RawArgShape
match ParameterShape and first-order TypeValueId compatibility
check body-entry policy admits Meta
invoke primitive or type-to-type construction interpreter
produce MetaValue / GraphDelta / Diagnostic
compute canonical meta instance key
install NamespaceDelta atomically
return object with explicit return-object policy
```

## 4. Product Semantic Normalization Must Become a Precondition Contract

Existing public docs define surface product / pipe / call binding. v0.8 needs an
additional semantic-layer `ProductObject` normalization. This rule must not be
conflated with surface normalization.

Core rules:

```text
flatten crosses Product nodes.
flatten does not cross Expression nodes.
order is preserved.
Unit is preserved.
provenance is preserved.
```

Definition:

```text
NF_P(P) = ordered sequence of product atoms

NF_P((x1, x2, ..., xn))
  = concat(NF_item(x1), NF_item(x2), ..., NF_item(xn))

NF_item(Product p)
  = NF_P(p)

NF_item(Expression e)
  = [e]

NF_item(Unit)
  = [Unit]
```

Examples:

```text
((P, e), e)   -> (P, e, e)
((e, e), e)   -> (e, e, e)
((e, P), e)   -> (e, P, e)
(e, (P, e))   -> (e, P, e)
```

If `P` is still a product, continue flattening until the product spine contains
no exposed Product child nodes.

"Exposed P" means a Product child node directly exposed in product context. A
product already absorbed by an expression/call node is not exposed. Flattening
does not cross expressions.

Therefore:

```text
((a, b), c)
=> (a, b, c)

(a, (b, c))
=> (a, b, c)

((a |> f), (b, c))
=> (a |> f, b, c)

((a, b) |> f, c)
=> ((a, b) |> f, c)
```

Incorrect rule:

```text
((a, b) |> f, c)
=> (a, b, f, c)        // forbidden
```

This is forbidden because `(a, b) |> f` is an Expression barrier.

However, a call may locally normalize its own source product:

```text
Call(source = ((a, b), c), target = f)
=> Call(source = (a, b, c), target = f)
```

This is not crossing an Expression from the outside. It normalizes the source
product inside that call.

`ProductObject` must not silently drop `Unit`:

```text
((a,), b)
=> (a, Unit, b)

((), a)
=> (Unit, a)
```

Only a future explicit parameter expectation / extraction rule may eliminate
`Unit`. Product semantic normalization itself does not.

## 5. Minimum ProductObject / ArgProductShape Requirements

Before entering meta invocation / generic type constructors, a normalized
product must be converted into semantic objects. The recommended minimum object
layers are:

```text
NormProduct
  -> ProductObject
  -> FlattenedProductObject
  -> ArgProductShape
  -> per-element RawArgShape
```

`ProductObject` must preserve at least:

```text
original normalized product
flattened atom sequence
arity
unit positions
nested product provenance
source spans / origins
material role
```

`FlattenedProductObject` invariants:

```text
ordered sequence of atoms
no direct Product atom remains
Expression nodes are opaque
Unit is preserved
left-to-right order preserved
provenance preserved
```

`ArgProductShape` is input to candidate preparation. It is not a type-checker
result and not a runtime ABI argument list.

Each atom forms one `RawArgShape`, marking at least:

```text
is value?
explicit pass mode?
known first-order TypeValueId?
rank/type object?
namespace object?
meta object?
pattern object?
residual/runtime expression?
origin/provenance
```

Parameter matching for type-level generic callables must not trigger automatic
argument passing. Type objects, rank objects, namespace objects, meta objects,
and pattern objects are not value arguments and must not receive inserted
`copy/ref/share/move/in` actions.

## 6. Symbol / Place / TypeValue Must Not Be Merged

Any generic type meta implementation must preserve:

```text
SymbolId
PlaceId
TypeValueId
```

Meanings:

```text
SymbolId    = symbol object identity in the namespace graph
PlaceId     = location identity that can be bound, updated, injected into, or opened for a NamespaceDelta
TypeValueId = canonical type value identity
```

Forbidden:

```text
TypeValueId equality implies PlaceId equality
SymbolId equality implies TypeValueId equality
alias creates fresh writable place
type-value equality grants write permission
```

Semantics of `let T: type = uint8`:

```text
symbol(T) = fresh symbol
place(T) = fresh writable place at current lexical level
type_value(T) = type_value(uint8)
```

Semantics of `let T === uint8`:

```text
alias(T) = uint8
value(T) = value(final_target(uint8))
place(T) = place(final_target(uint8))
```

`let f::T = ...` uses a place judgment, not a value judgment. Namespace
injection is a place operation, not a type-value operation.

## 7. AliasChain and Generic Arguments

Generic type-style meta-functions must support alias-aware type argument
evaluation, but usage must be distinguished.

At least three query modes must be distinguished:

```text
type-value evaluation:
  follow AliasChain to final value if alias semantics requires it

callable lookup:
  policy-aware symbol resolution; alias transparency must be explicitly specified

injection-place target:
  follow AliasChain only according to injection-place semantics, then check writability
```

Do not encode alias transparency as a global boolean. Alias behavior differs in
type-value position, callable target position, and injection-place position.

`AliasChain` must preserve at least:

```text
source symbol
forwarded target
final symbol
final value
final place
provenance chain
writable boundary
cycle detection
```

## 8. All Three Policy Planes Must Appear

Any meta callable / generic type constructor must distinguish:

```text
symbol visibility policy
body-entry policy
return-object policy
```

Meanings:

```text
symbol visibility policy:
  whether the current LookupEnv can see this symbol

body-entry policy:
  whether the current ExecutionEnv can enter the callable body

return-object policy:
  the policy of the object produced by the callable
```

Forbidden inference:

```text
symbol visible under PolicyEnv::Meta
=> callable body executable under Meta
```

Correct inference:

```text
symbol visible under PolicyEnv::Meta
=> meta pass may resolve / inspect / classify / residualize it

body-entry policy admits Meta
=> meta pass may enter / evaluate callable body
```

For ordinary generic type-style meta-functions, the minimum requirements are:

```text
callee symbol is meta-visible
callee payload is callable/meta-callable
body-entry policy admits Meta
return slot policy is explicit, e.g. meta + runtime let r: type
```

v0.8 does not require a complete policy lattice, but it must avoid
misrepresenting a meta function as a runtime function.

## 9. Generic Meta Instance Identity

Ordinary generic type-style meta-functions must have a canonical meta instance
key. Source text or normalized dumps are forbidden as the only identity.

Minimum key:

```text
callee function SymbolId
canonical argument product
canonical argument values, e.g. TypeValueId list
package identity
mount identity if relevant
build/config fingerprint
policy/export/provenance-relevant metadata
```

Examples:

```text
key(Vec(T)) =
  SymbolId(Vec)
  + ArgProductShape([TypeValueId(T)])
  + build/config fingerprint

key(Pair(A, B)) =
  SymbolId(Pair)
  + ArgProductShape([TypeValueId(A), TypeValueId(B)])
  + build/config fingerprint
```

Product canonicalization must happen before key computation. Exposed Products
must be flattened in order; Expression barriers must not be crossed; whether
`Unit` participates in the key must be defined by the `ProductObject` contract,
not guessed by implementers.

## 10. Meta Expansion and NamespaceDelta Atomicity

Any type-to-type meta construction must be installed through a
transaction-shaped result.

`MetaExpansionResult` / `MetaReductionResult` may contain:

```text
replacement_object
namespace_delta
diagnostics
provenance
cache_key_fragment
```

Rules:

```text
success:
  install replacement and namespace delta as one unit

failure:
  install no generated symbols
  retain diagnostics
  leave no partial generated namespace subtree
```

This rule applies to:

```text
struct
Vec(T)
Option(T)
Pair(A, B)
future type-to-type meta-functions
```

Forbidden:

```text
installing a half-generated type object and adding field functions later
leaving a partial type-associated namespace after failure
letting diagnostics depend on half-installed graph state
```

## 11. `struct` Must Migrate Toward Shared Meta Invocation

The current `struct` path may exist as a vertical slice, but new ordinary
generic type-style meta-functions must not expand that bespoke path.

Target direction:

```text
current struct / verify path
  -> shared meta invocation client

resolve callee
collect candidates
match ProductObject / ArgShape / TypeValueId
check body-entry policy
invoke primitive or interpreter
produce MetaReductionResult / MetaExpansionResult
install delta atomically
```

The `struct` payload may be primitive. Its lookup, argument parsing, policy, and
delta installation must not split into a separate mechanism from ordinary
generic type meta-functions.

## 12. Build Layer Must Enter Candidate Identity

The package/build layer is not external engineering scaffolding. It produces the
namespace graph and determines candidate search boundaries.

Every candidate object, meta instance, generated symbol, diagnostic, and cache
fragment must preserve sufficient build identity:

```text
package identity
namespace root
source root contribution
dependency mount
export surface
manifest/config fingerprint
provenance
```

Forbidden:

```text
candidate identity = raw path string only
generated type identity = callee name + source text only
external/internal visibility ignored
dependency mount identity ignored
```

## 13. Minimum Construction Order

Recommended stage order:

```text
1. Stabilize the ProductObject / ArgProductShape contract
2. Stabilize minimum TypeValueId / PlaceId / AliasChain objects
3. Stabilize callable signature objects for type-to-type meta-functions
4. Stabilize policy-aware lookup + minimum body-entry checks
5. Stabilize canonical meta instance keys
6. Stabilize NamespaceDelta atomic install for meta instances
7. Gradually migrate struct / verify to the shared invocation frame
8. Implement restricted type-to-type meta construction such as Vec(T) / Option(T) / Pair(A, B)
```

Do not skip steps 1, 2, 4, or 5 and directly implement `Vec(T)`.

## 14. Closed-Loop Checklist for Every PR / Patch

Any patch involving build, symbol, pattern, product, policy, meta, or generic
type construction must answer:

```text
Build:
  Does this object carry the needed package/mount/provenance/fingerprint slots?

Namespace graph:
  Does it work through SymbolObject / NamespaceDelta?
  Does it bypass the resolver?

Symbol identity:
  Does it distinguish SymbolId / PlaceId / TypeValueId?
  Does it incorrectly derive writable place from type equality?

Product:
  Does it go through ProductObject / ArgProductShape?
  Does it define exposed Product flattening?
  Does it preserve Unit / order / provenance?
  Does it avoid crossing Expression?

Pattern / candidate:
  Does it form RawArgShape / ParameterShape?
  Does it avoid hardcoding full overload early?

Policy:
  Does it distinguish symbol visibility / body-entry / return-object?
  Does it avoid the meta-visible => meta-executable error?

Meta invocation:
  Does it enter the shared invocation frame?
  Can it return MetaValue / GraphDelta / Diagnostic?
  Does it support future strict/partial boundaries?

Delta atomicity:
  Does failure install no partial generated subtree?

Alias:
  Is alias transparency distinguished by usage?
  Does it avoid manufacturing writable permission through aliasing?

Mechanical lowering:
  Does it avoid putting pass mode into TypeValueId?
  Does it avoid inserting pass actions for non-value arguments?
```

If any answer is unclear, do not continue expanding implementation features.
First add the contract or placeholder slot.

## 15. Minimum Requirements for Agents Reading This Document

Before executing v0.8-adjacent tasks, an agent must absorb at least these rules:

1. Treat this file as a construction plan, not as a replacement specification.
2. In the work plan, explicitly list the closed-loop nodes involved in the task:
   build, symbol, product/pattern, policy, meta invocation, and delta.
3. Before proposing an implementation, state whether there is any risk of
   bypassing the namespace graph, `SymbolObject`, `ProductObject`, or policy
   planes.
4. If the current code lacks an object, do not replace it with an ad hoc
   substitute. Prefer creating a placeholder object, metadata slot, or
   contract-shaped API.
5. If a shortcut is temporarily unavoidable, mark it as an implementation
   limitation in code comments, test names, or docs. Do not present it as
   language semantics.
6. Every new test must cover at least one invariant, not merely demonstrate a
   feature.
7. When handling generic type-style meta-functions, assume by default that
   `struct`, `Vec(T)`, `Option(T)`, and `Pair(A, B)` ultimately belong to the
   same shared meta invocation frame.
8. When handling product arguments, use the rule: exposed Products flatten in
   order, Expression barriers are not crossed, `Unit` is not dropped, and
   provenance is not dropped.
9. When handling policy, consider symbol visibility, body-entry, and
   return-object planes together.
10. When handling aliases, distinguish type-value evaluation, callable lookup,
    and injection-place target usage.

Minimum acceptance standard:

```text
An agent's plan must mention:
  ProductObject / ArgProductShape
  SymbolObject
  TypeValueId / PlaceId / AliasChain
  policy visibility / body-entry / return-object
  canonical meta instance key
  NamespaceDelta atomicity
  no-bypass rule
```

If these terms do not appear, the plan has not absorbed this construction
document.
