# v0.8 Semantic Spine: Symbolic Construction Values and Extraction

**Status: Future design narrative entry point. Not current implementation behavior.**
**This document defines the positive semantic model. It is the recommended first read for v0.8 meta construction.**

---

## 1. Values first

A call returns one value. Always.

A type-constructor call also returns one value:

```text
(int)Vec::std
```

This is not a `TypeValueId`. It is a symbolic construction value — the value
formed by the construction expression `(int) |> Vec::std`. Type computation is
only one projection of that value, not its definition.

## 2. The hourglass

Every construction value is a waist point:

```text
        extraction pattern space
              ↑
              |
        construct / extract
              |
              ↓
          value point
              |
              ?
              ↓
        next extraction view
```

Upward: the value participates in a constructor-specific isomorphism
(`construct_C(P) = v`, `extract_C(v) = P`).

Downward: `?` enters the value's currently exposed extraction view.

## 3. `?` enters the next extraction view

`?` does not mean "destructure." It means "enter the next extraction view."
Destructuring happens only if that view is product-shaped.

```text
leaf?   -> leaf          (idempotent)
P?      -> split(P)      (expose product)
C(P)?   -> extraction via constructor's declared interface
```

`(int Vec::std)? == int Vec::std` — it is a single return value, leaf in
ordinary value context. Extracting the type parameter requires a rank-pattern
context, not bare `?`.

Equality never inserts `?`. Binding context may insert one pattern-directed `?`.

## 4. Single return and product return are one mechanism

```text
let t = () |> single_return;
t? == t;                   // leaf, ? is idempotent

let t = () |> two_return;
t? != t;                   // ? enters product view
let a, b = t;              // binding may insert one ?  → `let a, b = t?`
(a, b) == t;               // false — equality never inserts ?
```

Named construction values expose their named extraction shape, not a bare product:

```text
val? == (a a::t, b b::t);         // true
(a a::t, b b::t) == val;          // false
(a a::t, b b::t) t == val;        // true — constructor restores the value
```

## 5. Meta-functions return values, not TypeValueId

```text
r = t   => generated construction value
r === t => forwarded existing value
```

These are not implementation variants of the same return behavior. The
meta-function body returns by ordinary block return rules; the difference
is the value semantics of the return slot.

## 6. Invocation returns a value; binding installs it

The invocation layer produces a value (`GeneratedConstructionValue`,
`ForwardedValue`, `Residual`, or `Diagnostic`).

The expansion/binding layer installs that value into the graph/world state
(`NamespaceDelta`, declared symbol binding, extraction interface exposure,
return-object policy application).

`IdentityType` proves invocation plumbing. It does not prove generative
type-to-type construction.
