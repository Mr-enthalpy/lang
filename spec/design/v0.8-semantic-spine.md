# v0.8 Semantic Spine: Symbolic Construction Values and Extraction

**Status: Future design narrative entry point. Not current implementation behavior.**
**This document defines the positive semantic model. It is the recommended first read for v0.8 meta construction.**

---

## 1. Values First

An evaluation returns one result object, whose normal form may be an ordinary
value `e` or a product value `P`:

```text
e:
  ordinary non-product value point

P:
  product value / product normal form
```

A type-constructor call returns a symbolic construction value:

```text
(int)Vec::std
```

This is not a `TypeValueId`. It is the value formed by the construction
expression `(int) |> Vec::std`. Type computation is only one projection of that
value, not its definition.

**Notation.** This document uses `(int)Vec::std` as the construction value form
and `(int) |> Vec::std` as the pipe/call skeleton from which it is derived. In
hole-free rank-annotation assertion contexts, the spelling `int Vec::std`
denotes the corresponding construction-shaped value; this is not a separate
semantic object.

## 2. The Hourglass

Every construction value is a waist point:

```text
        extraction pattern space
              ↑
              |
        construct / extract
              |
              ↓
          value point e
              |
              ?
              ↓
        exposed extraction view e' or P
```

Upward: the value participates in a constructor-specific isomorphism
(`construct_C(P) = v`, `extract_C(v) = P`).

Downward: `?` enters the value's currently exposed extraction view.

## 3. `?` Enters One Extraction View

`?` does not mean "destructure." It means "enter one exposed extraction view."
Destructuring happens only when a pattern matcher consumes product normal form.

```text
leaf e:
  e? = e

non-leaf e:
  e? = peel_one_exposed_extraction_view(e)
  // result may be e' or P

product P:
  P? = P
```

This view may be constructor-declared, but bare `?` is not the
constructor-specific inverse extractor. The upward `construct_C` / `extract_C`
isomorphism and the downward `?` view transition are separate paths. Extracting
a type parameter (for example from `Vec::std`) requires the constructor-specific
extractor in a rank-pattern context, not bare `?`.

`((int)Vec::std)? == (int)Vec::std` — the outer parentheses around
`(int)Vec::std` are postfix-`?` binding, not a product. The construction value is
a leaf in ordinary value context, so `?` is idempotent.

Equality never inserts `?`. Binding context may request one pattern-directed
`?`.

## 4. Product Return and Non-Product Return

For product return:

```text
let P = () |> two_return;
P == (a, b);              // result is product normal form
P? == P;                  // product normal form has no shell to peel
let a, b = P;             // product binding consumes P directly
(a, b) == P;              // true
```

For non-product value `e`:

```text
let e = () |> single_return;
e? == e;                  // if e is leaf
let a, b = e?;            // error if e? is still leaf
(a, b) == e;              // false if e is not product normal form
```

For a non-leaf construction value `e` with an exposed product view:

```text
e? == P
P == e?                   // true
P == e                    // false
P A== e                   // true under named constructor / pattern A
```

`A==` is provisional notation for constructor/pattern mediated equality. It is
not ordinary value equality.

Binding may request `?` only as pattern-directed repair:

```text
let a, b = e              // may try `let a, b = e?` if direct binding fails
```

## 5. Meta-Functions Return Invocation Values, Not TypeValueId

```text
r = t
  -> GeneratedConstructionValue or GeneratedTypeDefinitionValue, depending on returned material

r === t
  -> ForwardedValue(TypeSymbol)
```

These are not implementation variants of the same return behavior. The
meta-function body returns by ordinary block return rules; the difference is the
value semantics of the return slot.

`TypeValueId` is a derived first-order type-value projection material. It is not
an invocation result and is not construction or type-definition identity.

## 6. Invocation Returns a Value; Binding Installs It

The invocation layer produces:

```text
MetaInvocationResult::Value(MetaInvocationValue)
MetaInvocationResult::Diagnostic(Diagnostic)
```

Successful value cases include `ForwardedValue`, `GeneratedConstructionValue`,
and `GeneratedTypeDefinitionValue`.

The expansion/binding layer installs that invocation value into the graph/world
state (`NamespaceDelta`, declared symbol binding, extraction interface exposure,
return-object policy application). `bind_meta_invocation_value_result` is the
materialization boundary.

`IdentityType` proves invocation plumbing. It does not prove generative
type-to-type construction.

## Implementation Substrate Note

The v0.8 build substrate represents return normal form explicitly:

```text
EvalResultNormalForm = ValuePoint(e) | Product(P)
```

The `question_view` operation is shape-level and pure:

```text
Product(P) -> Product(P)
leaf e     -> e
non-leaf e -> exposed view
```

Generated struct type materialization records a queryable named field extraction
interface for instances of the type. That interface belongs to the materialized
`TypeObject` metadata, not to the pure `GeneratedTypeDefinitionValue` invocation
value itself.

Binding may request one extraction view as pattern-directed repair. Equality and
ordinary expression evaluation do not request `?`.
