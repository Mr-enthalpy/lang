# Static Pattern Spaces and Extraction Chains

## 1. Purpose

This design unifies several mechanisms that are often treated separately:

```text
pattern matching
branch exhaustiveness
if / else
result-consumption checking
construction / extraction correspondence
explicit discard
overload candidate filtering
```

The language does not need a privileged `match` syntax core, and `if` / `else` do not need to be lexer keywords. The more primitive mechanism is this: a value may explicitly enter a pattern space; a pattern space may be extracted; extraction produces both residual input space and completed results; final results must be consumed. Exhaustiveness follows from these rules rather than from a separate built-in `match` checker.

This design is not an arbitrary set-theoretic pattern algebra. It does not make the compiler an automatic theorem prover. A pattern space is a static object generated by a limited set of canonical constructors and controlled meta-operations. The compiler checks whether the required structure and meta-operation exist, whether they apply, and whether they return an acceptable static object. If an operation cannot be performed under known structure, the program is rejected. The compiler does not perform arbitrary Boolean simplification, arbitrary set-difference solving, or global logical proof.

This file is the fuller, later pattern-space / extraction-chain design. The earlier pattern normalization / candidate-shape layer used to prepare meta-invocation candidates is a different, earlier layer, documented in `spec/future/pattern-normalization-and-first-order-overload.md`.

## 2. Core Principles

### 2.1 Pattern spaces are static objects, not arbitrary sets

A pattern is not merely a syntactic matching object. A pattern denotes a static pattern space.

At a point in an expression, the current value has a static pattern space `A`. An extraction pattern `S` attempts to select a recognizable subspace from `A`.

Notation such as:

```text
S ⊆ A
A - S
```

should be read as structural operations over static pattern-space objects, not as general mathematical set solving.

More precisely:

```text
Pattern spaces are generated by canonical constructors.
Extraction checks perform mechanical removal over known structures.
If removal can be performed, it is performed.
If removal cannot be performed under the known shape, the extraction form is not applicable.
```

The compiler does not prove arbitrary containment between pattern formulas. It does not turn the pattern system into a full Boolean algebra solver.

### 2.2 Construction and extraction may be isomorphic; call and extraction are not

A constructor creates a value shape. An extractor may destructure the corresponding value shape. Therefore:

```text
construction and extraction may be isomorphic
```

A call, however, is not construction. A call merely returns a value. Extraction operates on the returned value’s static pattern space, not on the call target or the call history. Therefore:

```text
call and extraction are not isomorphic
```

For example:

```text
((a P1), b P2)
```

In expression position, `P1` and `P2` are ordinary value-side names, call targets, or expression-chain material. They are not pattern names. Even if `a P1` returns a value that is later extractable by some constructor pattern, extraction sees the returned value’s static pattern space. It does not use `P1` as an extraction name.

### 2.3 Values and patterns do not implicitly convert into each other

A value is not a pattern. A pattern is not a value.

```text
Value must not implicitly become Pattern.
Pattern must not implicitly become Value.
```

A value enters pattern space only through an explicit bridge, such as postfix `?` or another explicit value-to-pattern operation. A pattern produces or exposes values only through explicit extraction, binding, passing, or returning.

An ordinary local name that “looks like a pattern” does not automatically become a pattern name.

### 2.4 Pattern-side name resolution is bounded

Pattern-side names are not ordinary call targets.

Ordinary call target lookup may come from an open value/function binding environment. Pattern-head resolution is bounded by the expected extraction interface of the current value or classifier.

A pattern head may come from:

```text
fields
variants
constructor tags
built-in extractors
built-in type or classifier extraction rules
pattern interfaces explicitly produced by meta-functions
explicit extractor declarations
```

A pattern head must not come from:

```text
ordinary local value bindings
ordinary function bindings
ordinary overloaded call targets
arbitrary local names that merely look like pattern heads
```

There is no fallback from pattern-head lookup to ordinary function-call lookup. If a pattern head cannot be found in the current bounded extraction interface, this is a no-matching-extractor or no-candidate error in a later checking phase.

### 2.5 Behavior comes from locatable code, not compiler oracle knowledge

Operations such as `+`, `-`, `?`, and closed control-pattern behavior are not compiler-intrinsic mathematical truths. They come from locatable, auditable meta-operation implementations in libraries or built-in packages.

The principle is:

```text
because an author wrote these meta-operations,
these type/pattern computations can occur
```

not:

```text
because the compiler intrinsically knows full set algebra
```

The core compiler should remain first-order, mechanical, and conservative. It asks whether the required structure exists, whether the required meta-operation is available, and whether the operation returns an acceptable static object. If not, it reports an error.

## 3. Pattern Spaces

### 3.1 Atomic pattern spaces

An atomic pattern space denotes a pattern object that is either not decomposable at the current phase or not yet expanded.

Examples include:

```text
int
string
none
T
Option::std
```

In pattern position, these are pattern-side materials, not ordinary expressions. Whether they resolve to valid patterns is decided later by bounded pattern checking.

### 3.2 Product patterns

Product patterns describe extraction of multiple components from a structure. They emphasize mechanical structural matching.

Example:

```text
(head first, tail second) pair::std
```

This extracts two nodes from the `pair::std` construction. Every required node must be written, or explicitly consumed with `_`.

For product patterns:

```text
missing field
incomplete structure
wrong construction shape
```

are hard extraction-shape errors. They are not residual branches.

Thus:

```text
product pattern failure = structural error
```

A product pattern is not a backtracking “try it and skip if it fails” mechanism.

### 3.3 Sum patterns

Since the language has product patterns, it also has sum patterns.

A sum pattern is written:

```text
P1 | P2 | ... | Pn
```

It denotes a value that may inhabit one of several mutually exclusive branches.

Examples:

```text
(T some) | none
(T ok) | (E err)
```

Here:

```text
T some
none
T ok
E err
```

are branch patterns, while:

```text
(T some) | none
```

is a sum pattern.

The top-level branches of a canonical sum pattern are mutually exclusive. This is what allows ordinary residual computation to be mechanical:

```text
A - S
```

can remove known branches from a known sum pattern. This is still structural removal over a static object, not arbitrary set difference.

### 3.4 `|` and `+`

`|` and `+` are distinct.

```text
| is a result form.
+ is a construction / reduction action.
```

`|` denotes an already-formed canonical choice space:

```text
int | string
some<T> | none
```

`+` is a meta-level pattern-combination operation. It attempts to combine two pattern spaces into a canonical pattern space:

```text
int + string  => int | string
int + int     => int
```

Not every `P + Q` must be legal. Some combinations may be defined as non-constructible and may reach a deleted / non-constructible branch.

In pattern position, `P1 | P2` denotes pattern-side canonical sum structure. It is not ordinary value-side operator lookup. In expression position, `a | b` may still be ordinary operator sugar; but in pattern, annotation, or extraction position, `P | Q` belongs to pattern-side material and must remain distinct from `NormExpr`.

## 4. Meta-Operations and Compiler Boundaries

Pattern-space operations are performed through meta-operations. `operator+`, structural removal, postfix `?`, and related operations may be meta-functions or controlled implementations in built-in packages.

They are subject to the ordinary restrictions of meta-functions:

```text
deterministic symbolic binding discipline
no globally observable side effects
fixed and bounded input/output arity
staged execution
no observable dependence on lookup or evaluation order
```

`operator+` may merge two patterns, absorb duplicates, or reject certain combinations. The compiler itself does not know that `+` means mathematical union or that `-` means mathematical set difference. It invokes visible, locatable meta-operation implementations and checks whether the result is an acceptable static pattern-space object.

This design is not a global proof system. Its computability comes from restricted meta-operations written by authors, not from the compiler’s global understanding of arbitrary pattern formulas.

## 5. Extraction

### 5.1 Atomic extraction form

The atomic extraction form is:

```text
A |> S { body }
```

where:

```text
A    current input pattern space
S    subpattern to extract
body closure executed when S is hit
```

The basic well-formedness requirement is:

```text
S ⊆ A
```

This means that `S` must be mechanically recognizable as an extractable subspace of `A`. If `S` cannot be recognized in `A`, the program is not “adding a new branch” to `A`; it is using an inapplicable extractor. A later pattern-checking phase should report a no-matching-extractor or no-candidate error.

### 5.2 Hit and skip

If the current value inhabits `S`, the body executes and works on the payload carried by `S`. If the current value does not inhabit `S`, the body does not execute, and the unmatched residual branch continues to flow.

Formally:

```text
value ∈ S  => body executes
value ∉ S  => body does not execute; value remains in A - S
```

This “skip” is not a separate control-flow primitive. It is the contextual interpretation of extraction failure.

Skip is not:

```text
exception
return
implicit default value
discard
```

In a sum-pattern extraction chain, skip means that the current arm did not hit and that residual branches continue.

In overload candidate lookup, skip means that a candidate’s extraction pattern is not applicable to the current call pattern space, so the candidate is removed from the candidate set. Candidate filtering must be side-effect-free, repeatable, and independent of lookup order; otherwise overload resolution would become observable evaluation.

### 5.3 Product pattern failure vs sum pattern miss

Product and sum patterns fail differently.

```text
Product patterns emphasize structural completeness.
Sum patterns emphasize mutually exclusive branch selection.
```

Product pattern failure is a structural error: missing fields, wrong shape, or required parts not explicitly handled.

Sum pattern miss is not an error if `S` is a valid subspace of `A`. It only means that the current value belongs to `A - S`.

Thus:

```text
product pattern failure = structural error
sum pattern non-hit     = residual propagation
```

## 6. The `Done` Layer

### 6.1 Purpose of `Done`

When extraction hits, the body result must not be mixed directly with unhandled residual input. If the body returns pattern space `D`, the transformation is not:

```text
A - S + D
```

The transformation is:

```text
A - S + (D)Done
```

`Done` separates two kinds of material:

```text
A - S     input residual not yet handled by the current extraction chain
(D)Done   result already produced by an executed closure body
```

This prevents later same-level extraction arms from automatically re-matching results that have already been produced.

### 6.2 `Done` isolates by default but is explicitly re-enterable

`Done` is not a completely hidden compiler marker. It is an isolation layer.

```text
Same-level extraction chains do not implicitly inspect Done.
Users may explicitly match, consume, pass, or return Done-wrapped results.
```

Equivalently:

```text
Done is not automatically re-entered.
Done is explicitly re-enterable.
```

This distinction is necessary. If `Done` were fully hidden, users could not manually close, debug, or reorganize completed branch results. If `Done` were automatically visible, later same-level arms could accidentally process completed results. The correct boundary is: isolated by default, explicit on request.

### 6.3 Return and result boundaries

`Done` is not eliminated after each extraction arm. It is handled at explicit result boundaries.

A return or result boundary performs two steps:

```text
1. reduce one visible Done layer inside the returned expression;
2. wrap the resulting value space in Done again, because the enclosing closure/function has now produced a completed result.
```

In shorthand:

```text
return_boundary(E) = Done(reduce_one_done_layer(E))
```

For example:

```text
(D)Done  --return boundary reduces once-->  D
D        --as the completed result of this boundary-->  (D)Done
```

This preserves two invariants:

```text
A closure can explicitly close an inner extraction chain.
The closure result is still isolated from the outer same-level extraction chain.
```

Therefore, a return boundary is not merely “Done elimination”. It is local reduction followed by re-wrapping.

Important invariant:

```text
Done is not eliminated while same-level extraction continuation is still processing input residuals.
```

### 6.4 `match` as an explicit closer

`match` may be a conventional library-level closer, not a built-in control-flow syntax.

Its core shape can be approximated as:

```text
match := (x: _) => { x }
```

The `_` parameter explicitly consumes the current pattern space. The body returns `x`.

Because return boundaries perform one local `Done` reduction and then re-wrap the result, `match` closes one extraction-chain layer and produces a completed result for the enclosing context.

Example:

```text
bool_cond?
  |> if   { then_body }
  |> else { else_body }
  |> match
```

After the two branches:

```text
(D_then)Done + (D_else)Done
```

Inside `match`, `_` explicitly consumes the current pattern space. Returning `x` reduces one inner `Done` layer:

```text
(D_then)Done + (D_else)Done
=> D_then + D_else
```

Then, because `match` itself returns a completed result to its enclosing context, the result is wrapped again:

```text
(D_then + D_else)Done
```

The subsequent `operator+` reduction must determine whether `D_then + D_else` is a valid result pattern.

A non-exhaustive example:

```text
bool_cond?
  |> if { then_body }
  |> match
```

After the `if` arm:

```text
else + (D_then)Done
```

Inside `match`, the return boundary attempts one local reduction:

```text
else + (D_then)Done
=> else + D_then
```

Now `else + D_then` must close under `operator+`. Since `else` belongs to the closed control-pattern space `if | else`, it cannot be added to an unrelated result pattern. The operation reaches a deleted / non-constructible branch and fails before the result can be re-wrapped.

Thus:

```text
match performs no special matching.
match is the identity closer for the current pattern space.
match uses `_` to force explicit total consumption.
match's return boundary reduces one Done layer, then re-wraps the result.
The following operator+ reduction performs closure / exhaustiveness checking.
```

## 7. Result Consumption

Expression results must not be silently discarded unless the final pattern space reduces to pure `void`.

```text
final pattern = void   => silent completion is allowed
final pattern != void  => result must be consumed
```

A non-`void` result must be:

```text
bound
passed to another expression / call skeleton
returned
used as the final result of an enclosing body
explicitly matched and discarded with `_`
```

`_` is not implicit ignore. `_` is an explicit consumption pattern.

```text
missing consumer => error
written _        => explicit consumption
```

This rule prevents unhandled residual pattern spaces from disappearing merely because an expression is used as a statement. Exhaustiveness follows from residual propagation and result consumption, not from a separate privileged `match` checker.

## 8. `_` and Bare Branch-Name Arm Sugar

`_` explicitly consumes pattern material. It means the user intentionally does not use the matched payload.

The language permits one very small, restricted, mechanical sugar:

```text
|> name { body }
=> |> (_ name) { body }
```

This matches branch `name` and explicitly consumes its payload with `_`.

This sugar is not general implicit discard. It does not mean all omitted payloads may be ignored. It is only a fixed branch-arm shorthand.

For example:

```text
|> if { body }
```

may elaborate in branch-arm position to:

```text
|> (_ if) { body }
```

The payload discard is still explicit in the elaborated structure.

## 9. Postfix `?` and if / else

`?` is a unary postfix operator. It implements explicit value-to-pattern conversion.

```text
value?
```

sends a value into its corresponding pattern-space construction process.

For a bool-like condition:

```text
bool_cond?
```

the operator produces a closed sum pattern space:

```text
if | else
```

Here `if` and `else` are not lexer keywords and not built-in control-flow syntax. They are mutually exclusive branch names in the pattern space produced by `?`.

Traditional if / else can therefore be expressed as an ordinary extraction chain:

```text
bool_cond?
  |> if   { then_body }
  |> else { else_body }
  |> match
```

Using bare branch-name sugar, this can be read as:

```text
bool_cond?
  |> (_ if)   { then_body }
  |> (_ else) { else_body }
  |> match
```

Thus ordinary conditional branching is composed from:

```text
postfix ? producing a closed sum pattern space
extraction arms handling mutually exclusive branches
Done isolating completed branch results
match explicitly closing the chain
operator+ reducing result spaces
result-consumption rules enforcing exhaustiveness
```

## 10. Closed Control-Pattern Spaces

The pattern space produced by:

```text
bool_cond?
```

namely:

```text
if | else
```

is a closed control-pattern space. By default and by design, it cannot be combined with unrelated pattern spaces through `operator+`.

For example:

```text
(if | else) + P  => error
P + (if | else)  => error
```

unless `P` is a valid subpattern or same-family reduction inside that closed control-pattern space.

In ordinary cases, `operator+` reaches a deleted / non-constructible branch.

This prevents an unhandled `if` or `else` residual from being silently absorbed into an unrelated result space. If an `else` branch is not handled, it cannot be erased by a user-defined ordinary `operator+`.

### 10.1 This is not compiler magic

The non-additivity of closed control-pattern spaces is not a compiler oracle rule. It is guaranteed by ordinary language and package rules.

First, the globally available built-in `operator+` is real package code. An IDE should be able to jump to the implementation and show the deleted / non-constructible branch for combining `(if | else)` with unrelated pattern spaces. The naked root package space without a preceding navigation layer is virtual; it is not an open user namespace.

Second, package namespaces cannot be arbitrarily reopened downstream. “Opening a namespace and adding an overload” means modifying the owning package’s code. A user cannot inject a new `operator+` candidate into another package from an unrelated location.

Third, there is no unrestricted unqualified lookup. ADL-like behavior exists only if an operator implementation explicitly forwards to another lookup path. A candidate appears because locatable code explicitly admits it, not because the compiler performs global associative search.

In summary:

```text
operator candidate sets are package-owned and explicitly routed
there is no ambient extension
there is no unrestricted lookup
there is no automatic ADL
```

Closed control-pattern non-additivity is therefore a result of package-owned operator implementation, not a built-in set-theoretic axiom.

## 11. Binding Context

Binding is not a branch-transforming construction. `let` has no residual channel into which unhandled branches can flow. Therefore, binding extraction must be total:

```text
static_pattern - S = ∅
```

Example:

```text
let x some = option_val
```

is legal only if the static pattern space of `option_val` is already mechanically known as:

```text
T some
```

If it is still:

```text
(T some) | none
```

then extracting `some<T>` leaves residual:

```text
none
```

but `x` binds only the payload of `T some`. It cannot carry the `none` branch. This is a compile-time binding-shape error.

By default, binding extraction is total consumption of the current static pattern space. If future rules allow a broader binding pattern to be narrowed to the current space, that must be explicitly specified. It must not rely on general set-containment inference.

## 12. Overload Candidate Filtering and Resolution

> **Formal specification**: `spec/future/overload-resolution-design.md`.
> This section is a summary overview; the formal document defines the
> lexicographic extraction-pattern specificity rule, the full 11-step
> resolution pipeline with set notation (C0–C10), and the judgment form.

Extraction participates in overload candidate filtering and resolution. A candidate may carry an extraction pattern. If that pattern is not applicable to the current call pattern space, the candidate is skipped.

### 12.1 Overload Set Construction

An overload set is built from the symbols visible in the current namespace context, filtered by the current policy environment. `export` controls cross-package boundary visibility: only `export`-bearing symbols are visible to other packages for overload construction. Internal overload resolution (within the owning package) ignores `export`.

Namespace-level symbol lookup produces the initial candidate set. This lookup is **per-policy-pass**, not merged: meta/compile symbol lookup and runtime symbol lookup are separate passes with separate overload sets. Meta passes resolve only `Meta`-bearing symbols; runtime passes resolve only `Runtime`-bearing symbols. They are not searched together in one overload set.

All candidate filtering operations must be:

```text
side-effect-free
repeatable
independent of candidate-checking order
globally non-observable
```

### 12.2 Overload Specificity: Extraction-Pattern Depth

Overload specificity is **extraction-pattern specificity**. Candidates are not ordered by declaration sequence, source-file order, or an ad-hoc conversion-rank table. They are ordered by how deeply their extraction pattern penetrates the unified construction-expression tree.

The unified construction-expression tree `E` is the canonical structural form of the call operand. Every node has a depth:

```text
depth(root) = 1
depth(child) = depth(parent) + 1
```

A candidate overload pattern `P` is matched against `E`. **Every node explicitly written as a match or discard in `P`** — variable bindings, constructor matches, explicit `_` discards — contributes its depth to the specificity score:

```text
score(P, E) = Σ depth(n)
              for every explicitly written match/discard node n
              of P after matching P against E
```

A larger score means the candidate is more specific:

```text
P₁ more specific than P₂  iff  score(P₁, E) > score(P₂, E)
```

**Examples.** A generic built-in:

```text
ref(self, t: type)
```

matches only an abstract type parameter — contribution is small.

A more specific candidate:

```text
ref(self, let <t: type> t ref: type) => delete
```

explicitly matches `t ref`, penetrating to the `ref` constructor layer. Its score is higher, so it outranks the generic `ref(self, t: type)`. Similarly:

```text
ref(self, let <t: type> t share: type) => delete
```

outranks the abstract `ref(self, t: type)`, so `ref(T share)` hits the delete overload first and does not fall through to the built-in.

Explicitly writing a match or discard matters. Binding a variable, matching a constructor, or writing `_` all assert that the user knows and requires that structure. A pattern that operates only at an upper abstraction layer is less specific than one that penetrates deeper.

### 12.3 Overload Resolution Ordering

When multiple candidates survive initial filtering, they are ranked by a fixed multi-pass pipeline. Each pass eliminates candidates or orders them. The pipeline is:

| Step | Operation | Outcome |
|---|---|---|
| 1 | **Policy filter** | Remove candidates whose policy prevents use in the current context. A `runtime`-only candidate is removed from a meta pass. |
| 2 | **Pattern + type matching** | Remove candidates whose extraction pattern and type signature are structurally inapplicable to the call operand. |
| 3 | **Body-entry policy partial order** | `compile` > `meta` > `runtime` within the body of the function. `compile`-body candidates are preferred because they operate on fully value-materialized arguments; `meta`-body candidates follow; `runtime`-body candidates last. |
| 4 | **Concept legality** | Reject candidates whose concept constraints are violated by the call site. |
| 5 | **Concept partial order** | A stable partial-order projection over concept relationships. More constrained concepts outrank less constrained ones. |
| 6 | **Extraction-pattern specificity** | Depth-based scoring (§12.2). Larger total depth contribution → higher priority. Excludes candidates with equal maximal score. |
| 7 | **First-order before instantiated** | A first-order candidate (no type-parameter instantiation needed) outranks a candidate that requires instantiation. |
| 8 | **Lifetime pre-condition check** | Reject candidates failing `pre` / `lifetime pre` origin-path matching. This matching is structurally analogous to extraction scoring. |
| 9 | **Uniqueness** | If zero candidates remain → error. If more than one candidate remains → ambiguity error. If exactly one → select it. |

Because there is no unrestricted lookup, candidates do not appear from nowhere. ADL-like behavior only appears through explicit forwarding code.

### 12.4 Relationship to Policy and Namespace

Overload set construction is gated on policy-aware namespace lookup (§12.1). Meta-looking passes see only `Meta`-bearing candidates; runtime passes see `Runtime`-bearing candidates. This ensures that symbol lookup order does not leak between passes — a meta pass never accidentally resolves a runtime-only overload.

`export` controls the cross-package visibility boundary for overload construction. Within a package, `export` is irrelevant to overload resolution. Across packages, only `export`-bearing symbols appear in the namespace-wide candidate pool. Full `export` access control is deferred to later phases; the overload resolution pipeline assumes the candidate set has already been filtered by namespace visibility.

## 13. The Status of `match`

This design does not require a built-in `match` keyword to obtain exhaustiveness.

Exhaustiveness follows from:

```text
closed sum pattern spaces
extraction residual propagation
Done isolation
explicit match closing
operator+ result reduction
no silent discard of non-void results
explicit _ consumption
non-additivity of closed control-pattern spaces with unrelated patterns
```

A conventional `match` may exist in the standard package. It is not privileged syntax, not a separate matching semantics, and not compiler-intrinsic control flow. It is an identity closing consumer:

```text
match := (x: _) => { x }
```

Its name is appropriate because it expresses intent: close the current extraction chain, require the current pattern space to be consumed, and expose the closed result to the enclosing context as a completed result.

Thus:

```text
match is an ordinary function / visible library-level entity
match works through its `_` parameter and return boundary
match does not introduce a second matching mechanism
```

## 14. Relationship to Normalized AST

Most of this design belongs to later pattern checking, type checking, meta-function execution, package ownership, and symbol-binding phases. It does not belong to Raw AST → Normalized AST normalization.

The Normalized AST stage should preserve the necessary boundaries:

```text
value-side material remains NormExpr
pattern-side material remains NormPattern
annotation remains AnnotationPattern
DeduceList declares holes
HoleRef and PatternName remain distinct
alias target remains EntityRef
operators remain unresolved structural targets
postfix ? remains operator-shaped syntax
_ remains pattern-side explicit consumer material
bare branch-name arm sugar remains only the fixed elaboration into (_ name)
```

Normalized AST must not implement:

```text
pattern-space construction
S ⊆ A checking
A - S removal
Done insertion
Done reduction
Done re-wrapping
operator+ meta-reduction
closed control-pattern checking
result-consumption checking
void-only silent completion
exhaustiveness checking
pattern-head resolution
overload candidate filtering
binding totality checking
package-owner / namespace coherence checking
ADL-like forwarding rules
```

Those belong to later phases.

The Normalized AST only needs to preserve enough structure for later phases to distinguish:

```text
NormExpr::Name("P")        value-side ordinary name
NormPattern::Name("P")     pattern-side unresolved material
NormPattern::Nav(...)      pattern-side unresolved navigation material
NormPattern::HoleRef("T")  DeduceList-bound hole
OperatorTarget("?")        postfix value-to-pattern syntax material
Pattern-side "_"           explicit consumption pattern
```

## 15. Non-Goals

This design is not an arbitrary set-theoretic pattern system.

It does not promise:

```text
arbitrary pattern containment is decidable
arbitrary set difference is computable
arbitrary pattern formulas can be Boolean-simplified
every two patterns have a least common upper bound
the compiler can automatically discover all available extractors
the compiler can automatically search all related namespaces
```

This design is also not a traditional keyword-based control-flow system. `if`, `else`, and `match` may exist as ordinary names or library conventions, but the core semantics does not depend on them as privileged keywords.

This design is not an implicit-discard system. `_` is explicit consumption. A missing consumer is an error. Only expressions whose final pattern space reduces to pure `void` may silently complete.

This design also does not include call history in extraction. A call returns a value. Extraction processes the returned value’s static pattern space. The call target itself is not an extraction name.

## 16. Summary

The core of the design is:

```text
1. Patterns denote static pattern spaces.
2. Pattern spaces are generated by canonical constructors and restricted meta-operations.
3. Product patterns handle structure; sum patterns handle mutually exclusive branches.
4. `|` is canonical sum-pattern shape; `+` is a meta-level reduction action.
5. Extraction transforms A into A - S + (D)Done.
6. Done isolates completed results by default but is explicitly re-enterable.
7. Return/result boundaries reduce one Done layer locally, then wrap the returned result in Done again.
8. match is an identity closing consumer, not built-in matching syntax.
9. Expression results must not be silently discarded unless they are pure void.
10. `_` is an explicit consumption pattern.
11. `?` is postfix value-to-pattern conversion; bool? produces the closed pattern space if | else.
12. Closed control-pattern spaces cannot be added to unrelated patterns.
13. This non-additivity is guaranteed by package-owned operator implementations, non-reopenable namespaces, and absence of unrestricted lookup.
14. Construction and extraction may be isomorphic; call and extraction are not.
15. The compiler is not a theorem prover. It invokes locatable, restricted, author-defined meta-operations.
```

Exhaustiveness is therefore not a privilege of special `match` syntax. It is an ordinary consequence of pattern-space residuals, `Done` isolation, explicit closing, result consumption, and closed-pattern reduction.
